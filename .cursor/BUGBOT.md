# SmartHouseholdAccountBook — Bugbot レビュー方針

このファイルは [Bugbot](https://cursor.com/docs/bugbot) が PR レビュー時に参照するプロジェクト固有のルールです。セキュリティと保守性を優先してください。

## 全体

- 変更の意図が diff だけでは読み取れない場合は、**なぜその変更が必要か**を指摘し、PR 説明やコメントでの補足を促す。
- 大きな変更は **小さな PR に分割**できないか検討を促す（レビュー可能性とリスクの観点）。

## Spring Boot / セキュリティ

- **Spring Security** の設定変更（`SecurityFilterChain`、CORS、CSRF、認可ルール）は必ず重点確認。認可の抜け（他ユーザーのデータへのアクセス）がないか見る。
- **JWT / OAuth2** 周り: トークン検証の欠落、誤った audience/issuer、ログにトークンや PII を出していないか。
- **入力検証**: Controller の DTO と Bean Validation の整合、境界値と null。

## Frontend / Next.js セキュリティ

- ユーザー入力や API 応答を HTML として挿入していないか（`dangerouslySetInnerHTML` 等）。必要時はサニタイズがあるか確認する。
- `NEXT_PUBLIC_*` 経由で秘密情報を公開していないか。公開前提の設定値のみを許容する。
- 認証情報（JWT など）を不要に長期保存していないか。保存先と漏えい時影響（XSS 含む）を確認する。
- 外部 URL への遷移やリダイレクトでオープンリダイレクトの余地がないか確認する。

## Frontend / データ取得と状態管理

- API 失敗時に適切なエラー表示があるか。ローディング・空状態・失敗状態が実装されているか確認する。
- 更新系処理（作成/編集/削除）で二重送信防止（ボタン無効化・排他制御）があるか確認する。
- クライアント入力チェックだけに依存していないか。サーバー側検証を前提にした設計か確認する。
- 不要な再フェッチや再レンダリング（依存配列ミス、過剰 state 更新）がないか確認する。

## データベース / JPA

- **N+1** や意図しない eager ロード、トランザクション境界の不整合がないか。
- **生 SQL / ネイティブクエリ** がある場合は SQL インジェクションの余地がないか。

## 依存関係

- `pom.xml` やフロントのロックファイル変更時は、**不要な依存の追加**や既知の脆弱なバージョンへの更新がないか注意を促す。

## シークレットと設定

- API キー、パスワード、秘密鍵、`.env` の実値が **リポジトリに混入していないか**。ログ出力にシークレットが含まれていないか。

## テスト

- バグ修正や新規 API には **テストの追加または更新**があることが望ましい。欠けている場合は指摘する。

## パフォーマンス

- ホットパスでの不要な I/O、ループ内 DB アクセス、過大なレスポンス生成がないか。
- 大きな依存追加時はバンドルサイズ増加の影響を確認する。
- 一覧・グラフなどで不要な再計算がないか（`memo` / `useMemo` の適用余地）を確認する。
